Run [schema.sql](/smart-bandage/backend/supabase/schema.sql) in the Supabase SQL editor to create the necessary tables.

Heuristic details are documented in [temperature-heuristic.md](/smart-bandage/backend/docs/temperature-heuristic.md).

What it creates:
- `patients` for patient records
- `devices` for bandage/device assignment and wound-site baseline temperature
- `device_alert_settings` for per-device cold-spot and rebound thresholds
- `readings` for incoming temperature measurements
- `alerts` for warning/risk events generated by backend code
- `latest_device_readings` view for fast dashboard queries

Alerting behavior:
- The database stores raw readings and alert records
- The backend reads recent readings for a device after each insert
- The backend treats a temperature drop below baseline as a `warning` cold spot
- The backend treats a cold spot followed by a rapid rebound above baseline as `risk`
- If the heuristic crosses thresholds, the backend inserts a row into `alerts`

Important:
- RLS is enabled on all tables
- The current backend writes through the service role key, which bypasses RLS
- If you use the anon key from the frontend later, you will need explicit RLS policies
- If `device_alert_settings` has no row for a device, the backend uses default thresholds

Minimal setup example:

```sql
insert into public.patients (name, age, wound_type, admission_date)
values ('Jane Doe', 54, 'Surgical', current_date)
returning id;
```

```sql
insert into public.devices (device_id, patient_id, label, baseline_temperature_c)
values ('bandage-001', '<patient-uuid>', 'Right forearm', 34.2);
```

```sql
insert into public.device_alert_settings (
  device_id,
  window_minutes,
  min_rebound_readings,
  cold_spot_delta_c,
  inflammation_delta_c,
  rebound_rate_c_per_hour
)
values ('bandage-001', 120, 2, 0.5, 0.5, 0.6);
```
