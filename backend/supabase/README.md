Run [schema.sql](/Users/phongle/dev/smart-bandage/backend/supabase/schema.sql) in the Supabase SQL editor to create the necessary tables.

What it creates:
- `patients` for patient records
- `devices` for bandage/device assignment
- `device_alert_settings` for per-device alert thresholds
- `readings` for incoming temperature measurements
- `alerts` for warning/risk events generated by backend code
- `latest_device_readings` view for fast dashboard queries

Alerting behavior:
- The database stores raw readings and alert records
- The backend reads recent readings for a device after each insert
- The backend computes temperature change over time in API code
- If the heuristic crosses thresholds, the backend inserts a row into `alerts`

Important:
- RLS is enabled on all tables
- The current backend writes through the service role key, which bypasses RLS
- If you use the anon key from the frontend later, you will need explicit RLS policies
- If `device_alert_settings` has no row for a device, the backend uses default thresholds

Minimal setup example:

```sql
insert into public.patients (name, age, wound_type, admission_date)
values ('Jane Doe', 54, 'Surgical', current_date)
returning id;
```

```sql
insert into public.devices (device_id, patient_id, label)
values ('bandage-001', '<patient-uuid>', 'Right forearm');
```

```sql
insert into public.device_alert_settings (
  device_id,
  window_minutes,
  min_readings,
  warning_delta_c,
  risk_delta_c,
  warning_rate_c_per_hour,
  risk_rate_c_per_hour
)
values ('bandage-001', 120, 3, 1.0, 2.0, 0.3, 0.6);
```
